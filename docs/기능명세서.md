## B. 데이터·인프라 (비AI · Phase 1)

### B-1. PostgreSQL + pgvector 셋업

- **유형**: 코드
- **설명**: Oracle 박스에 Postgres + pgvector 설치. 멈췄던 "새 DB"를 이걸로 완성. 기존 `mysql2` 사용처를 Postgres 클라이언트로 전환. 타입 매핑: `tinyint(1)`→`boolean`, `datetime`→`timestamptz`, `enum`→`varchar`+`CHECK`, `AUTO_INCREMENT`→`GENERATED BY DEFAULT AS IDENTITY`(시드 시 기존 id 보존 후 `setval`).
- **이관 스키마(인터널 → 봇)** — DDL은 `db/migrations/001_directory.sql`:
  - `member` — 전체 22컬럼(`password`만 제외). member_type/status/authority는 CHECK. email·student_number UNIQUE.
  - `track` · `team` · `team_map`(member·team FK) · `member_withdraw`(member FK)
  - `pr_thread`(←`thread`) — pr_link·ts·reviewer·writer·status. **`reminder_sent` 제외**(GitHub 네이티브 리마인더가 담당)
- **봇 신규 스키마(RAG·운영)** — Phase 2~4에서 추가:
  - `document_chunk` — id, source, source_url, project, content, embedding(vector), last_synced_at
  - `sync_cursor` — connector, last_edited_at
  - `memory` — id, content, author, tags, created_at, embedding
  - `audit_log` — id, actor, action, target, before, after, created_at

#### B-1.1 13개 테이블 처리 방침 (확정)

| 처리 | 테이블 |
|---|---|
| **이관(Phase 1)** | `member`·`track`·`team`·`team_map`·`member_withdraw`, `pr_thread`(←thread) |
| **보류**(아카이브 덤프 보존, 해당 기능 착수 시 이관) | `dues`(회비 원장 1435), `jobs`(임원 이력 29), `slack_channel`·`slack_message`(아카이브) |
| **폐기** | `club_room_reservations`(예약) |
| **스킵**(빈/미사용) | `slack_thread`·`thread_reviewers` |

> **PR 스레드 흡수(별도 기능 태스크)**: `thread` 데이터 이관 후, `INTERAL_BASE_URL`로 위임하던 `b-bot/pull-request/thread` GET/POST를 **봇 라우트로 재구현**하고 `prThread.ts`가 봇 자신을 보게 한다. `pr-merged`의 `getPRThreadInfo` 미사용(메시지 갱신 미작동)도 이때 수정. → 인터널 **API 의존**까지 절단.

### B-2. 멤버 데이터 1회 마이그레이션 + 검수

- **유형**: 코드
- **설명**: 현재 인터널 DB 멤버 데이터를 봇 Postgres로 씨앗 복사하며 stale 데이터를 검수·정리. 이후 봇이 멤버의 임시 주인.
- **진행**: 원본 = 라즈베리파이 도커 `bcsd-mysql`(`bcsdlab`, 13개 테이블). **덤프 로컬 확보 완료** → `backup/bcsdlab_*.sql`(PII 포함, `.gitignore`). 로컬 MySQL 적재·검수 완료(활동 멤버 45명).
- **시드 정정 항목(마이그레이션 스크립트에 반영)**:
  1. `member 2357 (김성민)` `is_active 1→0` (active=1 & deleted=1 모순 정정)
  2. `password` 컬럼 미이관 (아카이브 덤프엔 보존)
  3. 전체 230행 이관, 봇은 `is_active=1`만 조회
  4. 시드 시 **본인을 `authority=ADMIN`** 지정(현재 활동 ADMIN 0 → 첫 관리자)
  5. 활동멤버 `slack_id` 결측 4명 보완(Slack 로그인 위해)
- **수용 기준**: 봇의 모든 멤버 조회가 봇 Postgres를 바라보고, 인터널 DB 의존성이 제거된다.

- **유형**: 코드
- **설명**: 코인 생태계(백엔드 API·안드·iOS·어드민·코넥트) + BCSD 사이트(홈·블로그·영양사 프론트) 레포의 README 인덱싱.

### C-2. Notion 커넥터 — ✅ 구현됨 (루트 트리 크롤)

- **유형**: 코드
- **설명**: 인수인계 문서·회의록을 수집.
- **구현 현황(2026-06-21)**: **루트(`Regular`) 트리 재귀 크롤 + 접근가능 DB 자동발견(B)** 병행. 컬럼/토글 재귀 + `link_to_page` 추적 + `child_database` 쿼리 + **`/search`(object=database)로 트리가 못 닿는 DB까지 발견**. **본문 있는 페이지만** 색인(빈 DB 행=출석부는 0청크 자동 스킵), 추적 DB 자동 스킵(앞 12개 빈 본문이면 중단). 이미 색인된 `source_url`은 스킵(재임베딩 방지). 배포 시점 자동 색인(startup) + 주간 cron(`sync:notion`).
- **핵심 발견(접근 모델)**: Notion에서 **페이지 연결이 하위 DB로 상속됨**(BackEnd 페이지 연결로 접근가능 DB 200→558 확인). 즉 *필요한 영역 페이지만 통합에 연결*하면 그 안 DB 전부 자동 색인(회원관리 등 PII 영역은 연결 안 하면 안전). 단 연결(공유)은 Notion 권한이라 코드가 못 뚫음.
- **삭제 안전화(중요)**: 긴 크롤(3~5h)은 일시에러 불가피 → 단일 크롤 seenUrls 기반 전체삭제는 **불완전 크롤 시 멀쩡한 콘텐츠 오삭제 위험**(실제 1회 발생, 7,125청크 손실). 수정: 삭제는 **`에러 0 + 완전 크롤`일 때만**, 불완전이면 스킵+로그. `nfetch`는 429/5xx/네트워크 5회 재시도.
- **남은 개선**: ① **증분 발견**(`last_edited_time` 내림차순 Search → 변경분만, 주간 ~수 분) ② **Notion SDK** (타입 + 신모델 `/data_sources` 지원). 현재 코퍼스 notion ~31K청크/4.2K문서.

## D. AI 읽기 — RAG·요약·분석

### D-1. 온보딩/도메인 Q&A (Phase 2 · 첫 PoC) — ✅ 구현됨

- **유형**: AI-읽기
- **설명**: 신입이 `!질문 env 어디 둬요?` / `!질문 코인 배포 어떻게?`처럼 질문하면, 인덱싱된 문서를 검색해 **출처와 함께** 답변.
- **입력**: 자연어 질문(`!질문 …`). **출력**: 답변 + 원본 링크.
- **데이터소스**: GitHub·Notion(✅) · Docs(C-3 예정).
- **구현 현황(2026-06-18)**:
  - **프로젝트 혼합 차단**: 질문을 레포 식별 임베딩(이름+desc+alias)과 비교해 확신 시 해당 레포로만 검색(사전 라우팅) + 폴백 시 상위 장악 레포로 한정(사후 보정). 소형 모델이 다른 레포와 섞지 못하게 검색 단에서 차단.
  - **헤딩 인식 청킹**: 청크에 헤딩 경로(breadcrumb) prepend → 표·목록 항목이 맥락과 함께 검색.
  - **스트리밍 UX**: 즉시 200 ACK → "검색 중…" → `chat.update`로 답 차오름. `keep_alive=30m`.
  - **검색 컨텍스트**: 원안 top-3 → **TOP_K 8**(정제·라우팅으로 노이즈 줄어 상향). 출처 필수 병기, "전원 공개" 문서만.
  - **타입 재랭킹(C-5)**: 회의록·재무 down-rank, 가이드·인수인계·readme 우선. 출처 4개 제한.
  - **안전**: 답변 본문에 URL 환각 금지(출처는 하단만), 문장 완결(`num_predict` 280).
- **모델**: 생성 **소형 로컬(B안, qwen2.5:3b)** — 품질·속도 부족 시 **A(Haiku)/C** 로 전환(`baseURL`+`modelId` 한 줄). 임베딩 bge-m3(로컬).

### D-2. 에러 자동 트리아지 — 🚫 보류(deferred)

- **상태**: **현재 범위에서 제외.** Glitchtip(백엔드 스택트레이스 소스)이 제거됨. 다만 `api/error-notice`(프론트 클라이언트 에러)는 살아 있어 *프론트 한정* 트리아지 입력원은 존재.
- **(참고) 원안**: 에러 웹훅의 스택트레이스를 분석해 *추정 원인 + 관련 코드 위치 + 담당자 추정*을 알림에 덧붙이고, 멤버 DB로 담당 트랙/작성자 멘션. (비동기/배치라 로컬 속도엔 무관했음)
- **재개 조건**: 백엔드 에러 소스(Sentry·앱 로그 등) 도입 시, 또는 `error-notice`의 프론트 에러만 우선 대상으로 한정 재설계.

### D-3. PR 리뷰 요약 — 🚫 보류(deferred)

- **상태**: **현재 범위에서 제외.** 추후 재검토.
- **제외 사유**: diff 추론은 로컬 7B와 Claude의 품질 격차가 가장 큰 작업이라 **유일하게 유료 API 의존이 필요했던 기능**. 정기 비용 $0 목표와 충돌하여 보류한다.
- **(참고) 원안**: `review-request` 웹훅 수신 시 diff를 요약(변경점·리스크·리뷰 포인트)해 PR 스레드에 코멘트, 기존 `postPRThreadInfo` 활용.
- **재개 조건**: ① 유료 API 예산이 생기거나, ② 로컬 코더 모델(Qwen2.5-Coder 등) 품질이 충분하다고 판단되거나, ③ 사람이 수동으로 요청할 때만 1회성 유료 승격하는 형태로 한정 도입.

### D-4. 회의록 요약 (Phase 3)

- **유형**: AI-읽기
- **설명**: (a) 음성→텍스트(whisper.cpp, 배치) → (b) 요약(결정사항·액션아이템·담당자) → 채널 게시. 기존 Google Meet 연동과 연계.
- **안전장치**: 배치(비실시간), Whisper는 `small`~`medium`.

### D-5. 주간 활동 요약 (Phase 3)

- **유형**: AI-읽기
- **설명**: 주간 활동 통계를 "이번 주 하이라이트" 문장으로 생성·게시. (기존 `!상태창` 주석 로직을 집계 소스로 부활)
- **트리거**: cron(예: 매주 월 9시), 배치.

## E. AI 알아듣기 — 자연어 행동

### E-1. 자연어 명령 라우팅 (Phase 4)

- **유형**: AI-알아듣기
- **설명**: 멘션 자유 텍스트를 의도 분류해 적절한 핸들러로 라우팅(기존 정규식 `!명령` 대체/보완).
- **안전장치**: AI는 의도/엔티티 추출까지만, 실행은 코드.

### E-2. 자연어 멤버 관리 (등급 변경) (Phase 4)

- **유형**: 혼합 (AI 판단 + 코드 실행)
- **설명**: "이준영을 레귤러에서 비기너로 바꿔줘" → `change_member_type(name, new_type)` 도구 호출로 번역 → 코드가 실행.
- **흐름**:
  1. (AI) 의도+엔티티 추출 → 도구 호출
  2. (코드) **권한 확인**(요청자 member_type/ADMIN)
  3. (코드) **식별**: 멤버 DB 대조, 동명이인이면 후보 제시
  4. (코드) **confirm**: Slack 버튼 "REGULAR→BEGINNER 변경할까요? [확인][취소]"
  5. (코드) 확인 시 `UPDATE member ...` (파라미터 바인딩)
  6. (코드) **감사 로그** 기록 → 완료 메시지
- **노출 도구(좁게)**: `change_member_type`, `find_member`, `save_note`, `recall_note`. raw SQL 권한 금지.
- **모델**: B안에선 **소형 로컬**로 의도+엔티티 추출(자유 생성이 아니라 도구 인자만 뽑으면 되어 부담 작음). 오작동은 **confirm 버튼**이 1차 방어. 추출 신뢰도가 부족하면 **이 경로만 A(Haiku)로 선택 승격**.
- **전제**: B-2(멤버를 봇이 소유)가 완료되어야 안전.

### E-3. 메모리 저장/회수 (Phase 4)

- **유형**: 혼합
- **설명**: "이거 저장해놔" → `save_note(content, author, tags)` 저장(임베딩) → 나중에 "그거 뭐였지?" → `recall_note`로 회수(출처: 작성자·일시).
- **데이터소스**: `memory` 테이블.
- **안전장치**: 사용자 생성 지식이므로 작성자·일시 표기, 동일 신뢰/최신성 규칙.

## 의존성 요약

```text
A(기반)  ──▶ B(DB·보안) ──▶ C(커넥터) ──▶ D-1(온보딩 RAG)
                  │                        │
                  ├─▶ E-2(멤버 쓰기)        ├─▶ D-4(회의록) D-5(주간)   [보류: D-2 에러·D-3 PR]
                  │                        └─▶ E-1(라우팅) E-3(메모리)
                  └─▶ G(Nuxt 어드민) ……… B 직후 착수 가능, RAG와 병렬 (같은 Postgres)
```

- B-2(멤버 소유) 미완료 시 E-2(자연어 등급변경)는 착수 금지.
- C(커넥터) 미완료 시 D-1(RAG)·D-2 코드 인덱스 의존 부분 보류.
- 비밀 분리(B-3)는 RAG 인덱싱(C) **이전**에 완료.
- **G(어드민)는 B(DB·멤버) 직후 착수 가능** — C/D(RAG)를 기다리지 않음. 봇과 같은 Postgres 공유라 별도 DB 없음.
