## 성능·운영 메모 (로컬 속도)

> 서버가 **2코어/12GB로 축소**되어 7B 동기 생성은 비현실적 → **현재 생성은 소형 로컬(3B/1.5B, B안)** 으로 $0 유지. 계획서 §2·§5.1·§7 참조.
>
> - **지연 민감(소형 모델로 속도 확보, 품질↓ 가능)**: D-1 RAG Q&A(최우선), E-1 라우팅, E-2 의도 추출, E-3 회수
> - **지연 무관(무난)**: D-4 회의록, D-5 주간, E-3 저장
> - **임베딩(bge-m3)·STT(whisper)는 로컬 유지** (배치라 2코어 무난, 외부 유출 없음)
> - **정책**: 지금은 **B(소형 로컬)** 로 완성 → **실사용 → 측정 → 품질·속도 부족 시 A(대면 Haiku)/C(하이브리드) 전환**(`baseURL`+`modelId` 한 줄). 선최적화 금지.

## A. 기반 정비 (비AI · Phase 0)

### A-6. Slack HTTP 웹훅 모드 확정

## B. 데이터·인프라 (비AI · Phase 1)

### B-1. PostgreSQL + pgvector 셋업

- **유형**: 코드
- **설명**: Oracle 박스에 Postgres + pgvector 설치. 멈췄던 "새 DB"를 이걸로 완성. 기존 `mysql2` 사용처를 Postgres 클라이언트로 전환.
- **스키마(초안)**:
  - `member` — id, name, slack_id, team_name, track_name, member_type(BEGINNER/REGULAR/MENTOR), is_active
  - `document_chunk` — id, source, source_url, project, content, embedding(vector), last_synced_at
  - `sync_cursor` — connector, last_edited_at (증분 동기화용)
  - `memory` — id, content, author, tags, created_at, embedding
  - `audit_log` — id, actor, action, target, before, after, created_at

### B-2. 멤버 데이터 1회 마이그레이션 + 검수

- **유형**: 코드
- **설명**: 현재 인터널 DB 멤버 데이터를 봇 Postgres로 씨앗 복사하며 stale 데이터를 검수·정리. 이후 봇이 멤버의 임시 주인.
- **진행**: 원본 = 라즈베리파이 도커 `bcsd-mysql`(`bcsdlab`, 13개 테이블). **덤프 로컬 확보 완료** → `backup/bcsdlab_*.sql`(PII 포함, `.gitignore`). 다음: 로컬 MySQL 적재 → 검수 → Postgres 스키마(B-1)로 이행.
- **수용 기준**: 봇의 모든 멤버 조회가 봇 Postgres를 바라보고, 인터널 DB 의존성이 제거된다.

### B-3. 비밀 분리

- **유형**: 코드
- **설명**: Drive의 API key·자격증명을 Infisical 등으로 이전, RAG/DB에서 영구 제외. 봇은 위치만 안내(값 노출 금지).
- **수용 기준**: 어떤 경로로도 봇이 자격증명 값을 출력하지 않는다.

## C. 지식 수집 커넥터 (코드 · Phase 2)

> 공통: 정규화(텍스트 + 메타데이터: source, url, project, 수정일) → 청킹 → bge-m3 임베딩 → pgvector. `project` 메타데이터로 스코프(코인/bcsd/공통). 출처 URL 보존.

### C-1. GitHub 커넥터

- **유형**: 코드
- **설명**: 코인 생태계(백엔드 API·안드·iOS·어드민·코넥트) + BCSD 사이트(홈·블로그·영양사 프론트) 레포의 코드·README 인덱싱. **push 웹훅**으로 증분 재인덱싱.

### C-2. Notion 커넥터

- **유형**: 코드
- **설명**: 트랙별 회의록, **인수인계 문서**(온보딩 최우선), 프로젝트 회의록을 블록→마크다운으로 수집. `last_edited_time` 기반 **cron 폴링** 증분.

### C-3. Google Drive(Docs) 커넥터

- **유형**: 코드
- **설명**: 기존 Google OAuth 활용. Docs를 텍스트로 export, Drive **changes API** + cron으로 증분.
- **제외**: 스프레드시트(→조회 도구), 인명부(→멤버 DB), 자격증명(→비밀 관리자).

### C-4. 동기화 스케줄러

- **유형**: 코드
- **설명**: Nitro `scheduledTasks`로 Notion/Drive 주기 폴링(예: 30분), 야간 재인덱싱. 동기화 커서는 `sync_cursor` 테이블.

## D. AI 읽기 — RAG·요약·분석

### D-1. 온보딩/도메인 Q&A (Phase 2 · 첫 PoC)

- **유형**: AI-읽기
- **설명**: 신입이 `@삐봇 env 어디 둬요?` / `코인 배포 어떻게?`처럼 질문하면, 인덱싱된 문서를 검색해 **출처와 함께** 답변.
- **입력**: 자연어 질문. **출력**: 답변 + 원본 링크.
- **데이터소스**: GitHub·Notion·Docs (C-1~3).
- **안전장치**: 검색 컨텍스트 **top-3·1000토큰 이내**(로컬 prefill 부담↓), 출처 필수 병기, "전원 공개" 문서만.
- **모델**: 생성 **소형 로컬(B안, Qwen2.5 3B/1.5B)** — 한국어 합성 품질·속도 부족 시 **A(Haiku)/C** 로 전환(`baseURL`+`modelId` 한 줄). 임베딩 bge-m3(로컬).

### D-2. 에러 자동 트리아지 — 🚫 보류(deferred)

- **상태**: **현재 범위에서 제외.** Glitchtip(백엔드 스택트레이스 소스)이 제거됨. 다만 `api/error-notice`(프론트 클라이언트 에러)는 살아 있어 *프론트 한정* 트리아지 입력원은 존재.
- **(참고) 원안**: 에러 웹훅의 스택트레이스를 분석해 *추정 원인 + 관련 코드 위치 + 담당자 추정*을 알림에 덧붙이고, 멤버 DB로 담당 트랙/작성자 멘션. (비동기/배치라 로컬 속도엔 무관했음)
- **재개 조건**: 백엔드 에러 소스(Sentry·앱 로그 등) 도입 시, 또는 `error-notice`의 프론트 에러만 우선 대상으로 한정 재설계.

### D-3. PR 리뷰 요약 — 🚫 보류(deferred)

- **상태**: **현재 범위에서 제외.** 추후 재검토.
- **제외 사유**: diff 추론은 로컬 7B와 Claude의 품질 격차가 가장 큰 작업이라 **유일하게 유료 API 의존이 필요했던 기능**. 정기 비용 $0 목표와 충돌하여 보류한다.
- **(참고) 원안**: `review-request` 웹훅 수신 시 diff를 요약(변경점·리스크·리뷰 포인트)해 PR 스레드에 코멘트, 기존 `postPRThreadInfo` 활용.
- **재개 조건**: ① 유료 API 예산이 생기거나, ② 로컬 코더 모델(Qwen2.5-Coder 등) 품질이 충분하다고 판단되거나, ③ 사람이 수동으로 요청할 때만 1회성 유료 승격하는 형태로 한정 도입.

### D-4. 회의록 요약 (Phase 3)

- **유형**: AI-읽기
- **설명**: (a) 음성→텍스트(whisper.cpp, 배치) → (b) 요약(결정사항·액션아이템·담당자) → 채널 게시. 기존 Google Meet 연동과 연계.
- **안전장치**: 배치(비실시간), Whisper는 `small`~`medium`.

### D-5. 주간 활동 요약 (Phase 3)

- **유형**: AI-읽기
- **설명**: 주간 활동 통계를 "이번 주 하이라이트" 문장으로 생성·게시. (기존 `!상태창` 주석 로직을 집계 소스로 부활)
- **트리거**: cron(예: 매주 월 9시), 배치.

## E. AI 알아듣기 — 자연어 행동

### E-1. 자연어 명령 라우팅 (Phase 4)

- **유형**: AI-알아듣기
- **설명**: 멘션 자유 텍스트를 의도 분류해 적절한 핸들러로 라우팅(기존 정규식 `!명령` 대체/보완).
- **안전장치**: AI는 의도/엔티티 추출까지만, 실행은 코드.

### E-2. 자연어 멤버 관리 (등급 변경) (Phase 4)

- **유형**: 혼합 (AI 판단 + 코드 실행)
- **설명**: "이준영을 레귤러에서 비기너로 바꿔줘" → `change_member_type(name, new_type)` 도구 호출로 번역 → 코드가 실행.
- **흐름**:
  1. (AI) 의도+엔티티 추출 → 도구 호출
  2. (코드) **권한 확인**(요청자 member_type/ADMIN)
  3. (코드) **식별**: 멤버 DB 대조, 동명이인이면 후보 제시
  4. (코드) **confirm**: Slack 버튼 "REGULAR→BEGINNER 변경할까요? [확인][취소]"
  5. (코드) 확인 시 `UPDATE member ...` (파라미터 바인딩)
  6. (코드) **감사 로그** 기록 → 완료 메시지
- **노출 도구(좁게)**: `change_member_type`, `find_member`, `save_note`, `recall_note`. raw SQL 권한 금지.
- **모델**: B안에선 **소형 로컬**로 의도+엔티티 추출(자유 생성이 아니라 도구 인자만 뽑으면 되어 부담 작음). 오작동은 **confirm 버튼**이 1차 방어. 추출 신뢰도가 부족하면 **이 경로만 A(Haiku)로 선택 승격**.
- **전제**: B-2(멤버를 봇이 소유)가 완료되어야 안전.

### E-3. 메모리 저장/회수 (Phase 4)

- **유형**: 혼합
- **설명**: "이거 저장해놔" → `save_note(content, author, tags)` 저장(임베딩) → 나중에 "그거 뭐였지?" → `recall_note`로 회수(출처: 작성자·일시).
- **데이터소스**: `memory` 테이블.
- **안전장치**: 사용자 생성 지식이므로 작성자·일시 표기, 동일 신뢰/최신성 규칙.

- `api/deploy`, `api/review-request`, `api/pr-merged`, `api/update-package`.
- `api/error-notice` — **유지**. 프론트엔드 범용 클라이언트 에러 리포터(`{url, error}` 수신 → KV 중복제거 → 채널 스레딩). Glitchtip과 무관.

### F-5. 인터랙션 처리

## G. 인터널 어드민 (Nuxt · 추후 = B 완료 이후)

> 봇 Postgres에 얹는 웹 GUI. **별도 DB 아님** — 봇과 같은 Postgres·같은 `audit_log` 공유. 멤버 데이터 **반납처**. Nuxt 서버=Nitro라 봇 자산(DB 어댑터·`member` 타입·플러그인 패턴·`useStorage`) 재활용, 새로 짜는 건 Vue 화면뿐. 착수 시 `packages/core` 추출 → pnpm workspace 모노레포 승격. 인증은 SpiceDB(Pi 상주) 또는 Slack OAuth. (계획서 §6.1·§6.2)

### G-1. 멤버 디렉터리 관리

- **유형**: 코드 (웹)
- **설명**: `member`/`team`/`track`/`member_type` 표·폼 CRUD, **일괄 등급·트랙 조정**, 탈퇴(`member_withdraw`), CSV import/export. 봇의 자연어 1건(E-2)과 같은 DB·`audit_log` 공유.
- **용도**: B-2 시드 후 stale 데이터 **검수·정리**를 GUI로 (SQL 대신).

### G-2. 감사·동기화 대시보드

- **유형**: 코드 (웹)
- **설명**: `audit_log`(봇 자연어 쓰기 이력) 열람, RAG **동기화 현황**(`sync_cursor`: 소스별 마지막 인덱싱·실패·재색인 버튼), 인덱싱 문서 목록·제외 항목 관리. ← Slack에선 불가능한 영역.

### G-3. (선택) 기존 테이블 관리 화면

- **유형**: 코드 (웹)
- **설명**: `dues`(회비) 납부 현황, `club_room_reservations`(동방 예약) 등 Pi에 이미 있는 데이터에 GUI만 부착.

```text
                  ├─▶ E-2(멤버 쓰기)        ├─▶ D-4(회의록) D-5(주간)   [보류: D-2 에러·D-3 PR]
                  │                        └─▶ E-1(라우팅) E-3(메모리)
                  └─▶ G(Nuxt 어드민) ……… B 직후 착수 가능, RAG와 병렬 (같은 Postgres)
```

- B-2(멤버 소유) 미완료 시 E-2(자연어 등급변경)는 착수 금지.
- C(커넥터) 미완료 시 D-1(RAG)·D-2 코드 인덱스 의존 부분 보류.
- 비밀 분리(B-3)는 RAG 인덱싱(C) **이전**에 완료.
- **G(어드민)는 B(DB·멤버) 직후 착수 가능** — C/D(RAG)를 기다리지 않음. 봇과 같은 Postgres 공유라 별도 DB 없음.
